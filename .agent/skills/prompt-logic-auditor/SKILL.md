# Skill: Prompt Options Logic Auditor (No-Overlap Options)

## 1) Цель скилла

Проверить сайт/конфиг генератора промптов изображений на логическую целостность:
- найти пересечения опций (семантические и функциональные),
- найти взаимные конфликты и невозможные комбинации,
- выявить дубликаты значений и “скрытые” синонимы,
- выявить некорректные зависимости/ограничения (requires/excludes),
- привести опции к полной, непротиворечивой структуре (таксономии),
- определить единственный источник правды: где, что и как должно храниться (schema).

Результат: понятная, строгая модель опций (категории, значения, ограничения), при которой пользователь не может выбрать пересекающиеся или конфликтные варианты, а система может валидировать и корректно собирать промпт.

---

## 2) Область проверки (Scope)

Скилл анализирует:
1) Все категории опций (например: Style, Lighting, Camera, Composition, Quality, Subject, Environment, Color, Mood, Renderer/Model-specific и т.п.)
2) Все значения в каждой категории.
3) Все зависимости:
   - requires (требует другое значение/категорию),
   - excludes (исключает),
   - implies (подразумевает),
   - overrides (перекрывает),
   - conflicts (противоречит),
   - mutual exclusivity (взаимоисключение).
4) Все функции/участки кода или конфигов, которые:
   - показывают/прячут опции,
   - применяют их к промпту,
   - нормализуют/склеивают/ранжируют,
   - маппят на модель (SDXL/MJ/Flux/и т.д. — если есть).

Скилл НЕ оценивает художественные качества промптов. Только структуру данных, логику выбора и непротиворечивость.

---

## 3) Входные данные (Inputs)

Скилл должен уметь работать с одним или несколькими источниками:

### 3.1 Конфиг опций (обязательно)
- JSON/YAML/TS/DB dump с описанием категорий и значений.
- Для каждого значения желательно:
  - `id` (уникальный),
  - `label` (что видит пользователь),
  - `prompt_token` (что вставляется в промпт),
  - `description` (смысл),
  - `tags` (семантика),
  - `weight/priority` (если применимо),
  - `scope` (модели/режимы/языки),
  - `constraints` (requires/excludes/…).

### 3.2 Логика фронта (желательно)
- правила отображения, disable/enable, hidden states,
- “preset” наборы, quick select, randomize.

### 3.3 Логика бэка (желательно)
- сборщик промпта,
- валидатор,
- нормализация токенов,
- правила “если выбрано X, то заменить Y”.

### 3.4 Примеры состояний (желательно)
- сохраненные пресеты,
- реальные payload-ы формы,
- sample промпты до/после применения опций.

---

## 4) Требования к анализу (What to check)

### 4.1 Уникальность и идентичность
Проверить:
- уникальность `id` по всему дереву,
- уникальность `prompt_token` в пределах одного контекста (или явные причины повторов),
- отсутствие дублей `label` (или явная причина: разные категории/контексты).

### 4.2 Семантические пересечения (Overlaps)
Найти пересечения, где разные опции выражают одно и то же:
- прямые синонимы: “cinematic” vs “film look”
- различия уровня: “soft lighting” и “diffused light” (если это один смысл в твоей системе)
- скрытые пересечения: одно значение включает другое (подмножество/надмножество)
- пересечения между категориями:
  - Style ↔ Color grading,
  - Lighting ↔ Mood,
  - Camera ↔ Composition,
  - Quality ↔ Model settings,
  - Art medium ↔ Renderer.

Для каждого overlap классифицировать:
- `duplicate` (дубликат — должен быть один источник),
- `alias` (алиас — можно оставить, но маппить на один token),
- `hierarchical` (иерархия: родитель/подтип),
- `orthogonal` (на самом деле не пересечение, можно сочетать).

### 4.3 Взаимоисключения и конфликты (Conflicts)
Проверить пары/наборы значений:
- физически невозможные: “underwater” + “desert”, если нет “fantasy”
- логически противоречивые: “black and white” + “vibrant color palette”
- конфликтующие команды модели: разные “quality” режимы
- конфликт времени/эпохи: “ancient Rome” + “cyberpunk neon city” (если не предусмотрен “mashup”)
- конфликт художественного медиума: “oil painting” + “photorealistic photo” (если ты считаешь их взаимоисключающими)

Типы конфликтов:
- `hard_conflict` (нельзя выбрать вместе),
- `soft_conflict` (можно, но предупреждение/понижение приоритета),
- `override` (одно значение должно явно перекрывать другое).

### 4.4 Зависимости (Dependencies)
Проверить корректность:
- requires: “anamorphic lens” требует “camera: film” или “cinematic” (если так задумано)
- excludes: “macro” исключает “wide aerial” (если так задумано)
- implies: “night” подразумевает “low light” (если система это делает)
- scope constraints: ограничения по модели/режиму (например, MJ vs SD, фото vs иллюстрация).

Найти:
- циклические зависимости,
- несовместимые requirements,
- зависимость на несуществующее значение,
- “хрупкие” правила, которые ломают UI (опции исчезают, но остаются выбранными).

### 4.5 Нормализация токенов (Token normalization)
Проверить:
- одинаковые токены с разным написанием (case, пробелы, тире),
- варианты через запятую/слеш (нужно ли нормализовать),
- конфликтующие префиксы/суффиксы (“highly detailed”, “ultra detailed”, “super detailed”),
- порядок токенов (важен/не важен для выбранной модели).

### 4.6 Полнота модели (Coverage)
Проверить:
- есть ли “дыры” в таксономии (например, есть 10 видов освещения, но нет базовых),
- есть ли категории “свалки” (misc) — признак плохой структуры,
- есть ли неиспользуемые значения (мертвые опции),
- соответствие UI: каждое значение должно иметь объяснение и прогнозируемый эффект.

## 5) Метод анализа (How to audit)

### Шаг 1: Сбор единой таблицы опций
Построить плоскую таблицу:
- category_id, category_label
- option_id, option_label
- prompt_token
- description
- tags/keywords
- constraints (requires/excludes/…)
- scope (модель/режим)
- source (файл/путь/компонент)

### Шаг 2: Автоматические проверки целостности
- уникальность id/token
- поиск exact-duplicate label/token
- поиск near-duplicate (Levenshtein/триграммы) по label/token
- группировка по общим ключевым словам в description/tags

### Шаг 3: Семантическая матрица пересечений
Сформировать матрицу:
- строки: значения
- колонки: значения
- тип отношения: duplicate/alias/hierarchical/conflict/orthogonal/unknown
Сфокусироваться на:
- внутри одной категории (mutual exclusivity)
- между соседними категориями (Style ↔ Color ↔ Mood, Camera ↔ Composition ↔ Lighting)

### Шаг 4: Анализ правил UI и сборщика промпта
Проверить:
- соответствуют ли UI-ограничения back-end валидатору,
- есть ли расхождения: UI разрешает, backend ломает или наоборот,
- порядок применения правил, overrides.

### Шаг 5: Предложение новой структуры (Refactor plan)
Собрать:
- список merge (что объединить),
- список split (что разделить),
- список move (что перенести в другую категорию),
- список alias (что оставить как синоним),
- список hard_conflict (что запретить),
- список soft_conflict (что предупреждать),
- список requires/excludes (что добавить/исправить),
- обновлённый schema + миграции пресетов.

---

## 6) Выходные артефакты (Outputs)

Скилл должен генерировать:

### 6.1 Отчет по проблемам
Для каждой проблемы:
- severity: blocker / major / minor
- тип: duplicate / alias_needed / conflict / bad_dependency / scope_mismatch / dead_option / unclear_definition
- где найдено (source path, id)
- пример конфликтной комбинации (если применимо)
- рекомендация исправления

### 6.2 Предложенная таксономия
- дерево категорий (уровни: Category → Subcategory → Option)
- правила взаимного исключения внутри категории
- правила пересечений между категориями
- единый словарь токенов + алиасы

### 6.3 Спецификация валидатора (Validator spec)
- список валидируемых правил (requires/excludes/overrides)
- поведение при конфликте:
  - блокировать выбор / автоснимать / показывать предупреждение
- детерминированность результата (одинаковый input → одинаковый prompt)

### 6.4 План миграции
- как преобразовать старые пресеты в новую схему
- как обработать удалённые/объединённые значения

---

## 7) Правила построения хорошей структуры (Design rules)

1) Один смысл — один источник правды:
   - если есть “cinematic”, то “film look” либо алиас, либо подтип, но не отдельная сущность без правил.

2) Категории должны быть ортогональны:
   - Style ≠ Lighting ≠ Camera ≠ Composition ≠ Mood
   - если значение смешивает 2 оси (“cinematic warm lighting”), разделить.

3) Внутри категории значения должны быть взаимоисключающими или явно комбинируемыми:
   - если комбинируемы — это должна быть не radio-группа, а multi-select, и наоборот.

4) Overrides должны быть явными:
   - если “black and white” перекрывает “vibrant colors”, это должно быть записано правилом, а не случайностью.

5) Scope должен быть первым классом:
   - если опция работает только для конкретной модели/режима, это должно быть частью schema.

6) UX-логика должна совпадать с backend:
   - UI не должен позволять то, что backend потом “чинит” молча.

---

## 8) Acceptance criteria (Критерии приемки)

Скилл считается выполненным, если:
- нет конфликтов severity=blocker без предложенного исправления,
- все дубликаты и алиасы перечислены и приведены к единой схеме,
- для каждой категории определено: single-select/multi-select + правила исключения,
- определен полный набор requires/excludes/overrides, без циклов и битых ссылок,
- сформирована финальная таксономия и валидатор-спека,
- предложен план миграции существующих пресетов.

---

## 9) Пример формата правил (Rules DSL, пример)

> Пример, как описывать правила (адаптируй под свою систему):

- mutual_exclusive:
  - category: color_mode
    options: [bw, vibrant_color, pastel]
- requires:
  - if: camera_lens=anamorphic
    then: style=cinematic
- excludes:
  - if: composition=macro
    not: composition=aerial
- override:
  - if: color_mode=bw
    remove: [palette=vibrant, palette=neon]
- scope:
  - option: quality=ultra
    allowed_models: [sdxl, flux]
    disallowed_models: [mj]

---

## 10) Как применять скилл (Operational steps)

1) Получить конфиги + список страниц/компонентов UI + сборщик промпта.
2) Сгенерировать таблицу опций.
3) Запустить проверки (uniqueness / duplicates / near-duplicates).
4) Составить матрицу пересечений и конфликтов.
5) Выдать отчет + новую структуру + спецификацию валидатора + миграцию.
