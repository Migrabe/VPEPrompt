## Design Context

### Users
Основные пользователи: соло-креаторы.
Контекст использования: быстрый запуск и итеративная донастройка промптов под разные модели генерации (Midjourney, DALL·E, SD, Flux, NBP и др.) в одном интерфейсе.
Job-to-be-done: собрать рабочий промпт быстро, без потери управляемости, и получить предсказуемый результат при смене модели.
Приоритет сценариев (в порядке важности): скорость -> точность -> консистентность -> креативный поиск.

### Brand Personality
3 слова: уверенный, премиальный, спокойный.
Тон интерфейса: профессиональный, дружелюбный и без визуального шума, с акцентом на простоту в использовании даже при большой глубине настроек.
Коммуникация в интерфейсе: краткие, прикладные формулировки; пояснение причин блокировок и конфликтов без "маркетинговой воды".
Анти-тон: игрушечный, кричащий, перегруженный визуальными эффектами.

### Aesthetic Direction
Тема: светлая + тёмная (паритетная поддержка обеих).
Текущая база проекта (iOS/macOS-inspired, карточный интерфейс, акцентные цвета) сохраняется, но решения должны быть функционально-прагматичными в первую очередь.
Анти-направление: перегруженные декоративные эффекты, агрессивные контрасты и визуальные паттерны, мешающие быстрому выбору параметров.
Цветовая семантика: основной акцент для первичных действий, зелёный для подтверждения/успеха, оранжевый для режимов внимания, красный только для деструктивных действий.
Локализация: базовый UI на русском с последовательной терминологией; английские названия допустимы для моделей/брендов и имен пресетов.
Доступность: базовый уровень (обычная) — видимый фокус, читаемые состояния disabled/active, отсутствие критически важной информации, кодируемой только цветом.

### Design Principles
1. Скорость без потери контроля: ключевые действия и пресеты всегда в зоне первого экрана, минимум лишних кликов и прокруток.
2. Точность через ясность: каждый конфликт или блокировка имеют видимую причину и читаемую подсказку "что изменить".
3. Консистентность результата: одинаковые паттерны поведения для переключателей, пресетов и режимов во всех моделях.
4. Креативность без хаоса: расширять набор стилей и пресетов, но не смешивать несовместимые режимы без явного предупреждения.
5. Простота как стандарт: сложные настройки раскрываются прогрессивно, а базовый поток остаётся понятным с первого запуска.

### Normalization Rules (/normalize)
1. Использовать существующие дизайн-токены (`--bg`, `--surface`, `--text`, `--accent-k`, `--accent-v`, `--accent-s`, `--green`, `--orange`, `--red`, `--radius`, `--radius-lg`) как основной источник цвета и формы.
2. Не вводить новые one-off паттерны кнопок, секций и тегов, если их можно собрать из текущих `option-btn`, `action-btn`, `section`, `mode-badge`.
3. Новые визуальные состояния обязаны иметь парные light/dark значения (без перекоса в одну тему).
4. Иерархия действий фиксированная: primary action -> secondary action -> destructive action.
5. Конфликтные режимы и блокировки должны выглядеть единообразно во всех секциях.

### Hardening Rules (/harden)
1. Любой пользовательский текст в контролах и подсказках должен корректно переживать длинные строки и смешанный RU/EN контент.
2. Состояния `loading`, `empty`, `error`, `offline` для ключевых действий (сборка промпта, перевод, улучшение, копирование) должны быть явно различимы.
3. Критически важные статусы нельзя кодировать только цветом; требуется дополнительный текст/иконка/лейбл.
4. Фокус-контур обязателен для интерактивных элементов, включая клавиатурный сценарий.
5. При конфликте параметров пользователь всегда получает понятное объяснение причины и следующего шага.

### Optimization Rules (/optimize)
1. Любые UI-изменения не должны увеличивать визуальную задержку основного потока (модель -> параметры -> результат -> копирование).
2. Избегать тяжёлых эффектов, влияющих на скролл и отклик (избыточный blur, сложные тени, частые repaint-операции).
3. Анимации использовать только для ориентации пользователя; при `prefers-reduced-motion: reduce` отключать несущественные эффекты.
4. Для нового контента вне первого экрана применять прогрессивное раскрытие вместо одновременного рендера всего объёма.
5. Любое изменение производительности проверять практикой: интерфейс должен оставаться отзывчивым на среднем ноутбуке/десктопе.

### Adaptation Rules (/adapt)
1. Поддерживаемые адаптивные пороги проекта: `1320`, `1200`, `1100`, `900`, `768`, `600`, `560`, `480` px.
2. На узких экранах приоритет сохраняется за основным рабочим потоком и быстрыми действиями, вторичные панели уходят в прогрессивное раскрытие.
3. Интерактивные элементы должны оставаться удобными для тач-ввода (минимально комфортная зона на мобильных).
4. Контент и контроли не должны ломать сетку при длинных названиях пресетов, моделей и подсказок.
5. Переход между desktop и mobile не должен менять смысловые паттерны (только форму представления).

### Clarity Rules (/clarify)
1. Базовый язык интерфейса: русский. Термины использовать последовательно: "Режим генерации", "Быстрые пресеты", "Редактируемые пресеты", "Сброс", "Копировать", "Компакт".
2. Тексты действий формулировать как конкретный глагол + объект, без расплывчатых CTA.
3. Ошибки писать по схеме: что произошло -> почему -> что сделать сейчас.
4. Подсказки должны объяснять практический эффект параметра, а не повторять его название.
5. Избегать жаргона без пользы и длинных абзацев в микрокопирайте; приоритет за краткостью и однозначностью.

### Definition of Done (UI)
1. Пользователь может собрать и скопировать валидный промпт или JSON без чтения документации.
2. При включении конфликтующих опций система явно показывает, что отключено и как вернуть совместимое состояние.
3. Light/Dark темы визуально равноправны: читаемость, контраст и иерархия не деградируют между темами.
4. Основной путь (выбрать модель -> собрать параметры -> получить итоговый промпт/JSON -> скопировать) не требует лишних возвратов по экрану.
5. В интерфейсе нет новых one-off стилей/паттернов, противоречащих существующей системе токенов и компонентов.
